Majeur Pas de reprise explicite d'un /migrate interrompu (le --resume d'Arbor manque) dxskills/legacy-upgrade/SKILL.md:66 🍎M

Arbor se reprend d'un Ctrl+C : `arbor --resume` recharge l'état et continue. Le kit a de fait les mêmes checkpoints — un commit à chaque gate vert, l'état dans migration/report.json — mais aucune affordance ne les lit. Une session interrompue en phase 4 oblige la suivante à deviner où reprendre : relire les artefacts, reconstituer la phase courante, risquer de rejouer une phase déjà verte. Les checkpoints existent ; le produit ne les exploite pas. ✅ RÉSOLU (2026-07-23, v1.8.0) : protocole de reprise dans SKILL.md (Scope variants — Resume : détection de migration/ + commits de porte nommant leur phase, artefacts en confirmation, annonce du point de reprise, ré-entrée à la phase suivante, une phase verte n'est jamais rejouée) ; « Resume, never restart » dans commands/migrate.md ; ligne « Session interrupted mid-pipeline » dans Common issues ; propriété « Resume » documentée au README.

→ Au lancement, /migrate détecte un dossier migration/ existant et le dernier commit de gate vert (message nommant la phase, règle 4), annonce « migration en cours détectée — reprise en phase N » et ré-entre à la phase suivant le dernier gate vert. Documenter ce comportement dans SKILL.md (Scope variants) et commands/migrate.md.

Majeur Phase 4 sans garde de convergence — la boucle de remédiation peut tourner à vide error-handlingskills/legacy-upgrade/SKILL.md:49 🍎S

Arbor borne toute exploration : max_cycles, max_turns par executor, politique de convergence. La phase 4 du kit (« drive diagnostics to zero ») est une boucle sans limite : sur un repo pathologique où les bulk-fixes créent autant de diagnostics qu'ils en résolvent, rien ne détecte la non-convergence — le pipeline consomme du budget agent sans signal d'échec, ce qui contredit la promesse « fast » du produit. ✅ RÉSOLU (2026-07-23, v1.8.0) : hard rule 9 « Remediation must converge » (compte d'erreurs suivi à chaque passe ; deux passes consécutives sans baisse = stop, retour à la dernière porte verte, blocage consigné avec les diagnostics restants groupés par id, décision au propriétaire) ; garde de convergence dans phase-4-remediate §4 ; ligne dédiée dans Common issues.

→ Règle de convergence en phase 4 : si deux passes consécutives de remédiation ne réduisent pas le compte d'erreurs, stop — retour au dernier commit vert, la situation est consignée dans le rapport comme blocage avec les diagnostics restants groupés par id, et l'humain décide. Une ligne dans les hard rules + une entrée dans Common issues.

Mineur Le chiffre-titre du kit (« 18 min ») est mesuré à la main, pas par l'outillage observabilityREADME.md:18 🍎S

Arbor horodate tout : run_stats, events.jsonl, rapport final généré. La colonne « Measured pipeline time » du README est l'argument commercial n°1 du kit — et c'est le seul chiffre du produit qu'aucun script ne capture. C'est contraire à la règle 7 du kit lui-même (les outils déterministes rendent les migrations reproductibles et comparables) : la preuve phare repose sur un chronomètre humain. ✅ RÉSOLU (2026-07-23, v1.8.0) : `phases[]` (début/fin/minutes par phase) dans migration/report.json, dérivé des commits de porte de la branche de migration (phase-6-verify §6 — un fait généré depuis git, jamais un chronomètre) ; carte « Chronologie du pipeline » dans report-dashboard.py avec total calculé ; contrat d'artefacts (SKILL.md) et README (« Measured time ») mis à jour ; test golden étendu (fixture `phases` + assertions minutes/total).

→ Horodater chaque phase dans migration/report.json (début, fin, durée par phase + total pipeline) au moment où le gate passe au vert ; report-dashboard.py affiche la ligne de temps par phase. La colonne du README devient un fait généré, cité depuis le rapport.

Mineur La rétropropagation des leçons est une culture, pas un contrat de phase 7 documentationskills/legacy-upgrade/SKILL.md:52 🍎S

Le cœur d'Arbor est la backpropagation : chaque expérience remonte « ce qui a marché / ce qui a échoué » vers les nœuds ancêtres, et l'arbre en hérite. Le kit vit déjà cela — la vague 2 a produit v1.3.2 (asset-inventory), la vague 3 a produit v1.4.1 (projets zombies) — mais rien ne l'exige : c'est la discipline de l'opérateur, pas une étape du pipeline. Un autre utilisateur livrera en production sans que ses leçons ne touchent jamais le kit, et le produit cessera d'apprendre. ✅ RÉSOLU (2026-07-23, v1.8.0) : règle 8 étendue — la phase 7 se clôt par l'entrée `lessons` de report.json (référence du changement appliqué au kit, ou « rien à apprendre de cette vague » explicite) et une vague sans entrée leçons est incomplète ; étape 9 du delivery-playbook (schéma `lessons[]` : strong/text/ref) ; carte « Leçons de la vague » dans le dashboard ; report-template mis à jour ; test golden étendu.

→ Étape formelle en phase 7 (delivery-playbook) : la migration se conclut par une entrée « leçons » dans migration/report.json — soit un renvoi vers un diff appliqué au kit (ligne Common issues, playbook, garde de script), soit un « rien à apprendre de cette vague » explicite. Le rapport final l'affiche ; une vague sans entrée leçons est incomplète.

Info L'audit de portefeuille est séquentiel — le dispatch parallèle d'Arbor s'y appliquerait dxcommands/migrate-audit.md:1 🍎M

Arbor dispatch ses executors en parallèle dans des worktrees isolés (RunExecutorParallel). L'audit de portefeuille du kit traite les apps l'une après l'autre, alors que les inventaires par app sont indépendants par construction (audit-inventory.sh produit un JSON par repo) et que la seule étape réellement transverse est la synthèse portefeuille. ✅ RÉSOLU (2026-07-23, v1.8.0) : paragraphe « Multiple apps: fan out » dans commands/migrate-audit.md — un sous-agent par app (inventaire + rapport d'app en parallèle, même discipline lecture seule / scripts du kit obligatoires), l'orchestrateur ne garde que la synthèse portefeuille. Aucun script modifié.

→ Documenter le pattern fan-out dans /migrate-audit : un sous-agent par app cible exécute inventaire + rapport d'app en parallèle, l'orchestrateur ne garde pour lui que la synthèse (matrice valeur/effort, ordre de migration). Aucun changement de script requis.

Info Consigner les non-adoptions : Idea Tree, modes d'interaction et novelty search sont hors périmètre simplicitydocs/backlog.md:1 🍎S

Arbor et le kit sont des produits opposés : Arbor explore un espace ouvert (métrique à maximiser, meilleure solution inconnue) ; le kit exécute un chemin connu vers une destination binaire (build vert, tests verts, prod vérifiée). Importer l'arbre d'hypothèses, les modes d'interaction (auto/direction/review) ou la recherche de nouveauté ajouterait des concepts sans améliorer une seule migration — et détruirait la propriété qui fait la valeur du kit : 18 minutes, déterministe, reproductible. Sans décision écrite, la tentation de « faire explorer » le pipeline reviendra. ✅ RÉSOLU (2026-07-23, v1.8.0) : section « Non-adoptions (décisions fermées) » dans docs/backlog.md — Idea Tree / recherche multi-hypothèses, ui.interaction_mode, novelty search alphaXiv, chacun refusé avec sa justification et une condition de réouverture explicite (changement de nature du kit, jamais une migration).

→ Entrée « non-adoptions » dans docs/backlog.md : Idea Tree / recherche multi-hypothèses, ui.interaction_mode, novelty search alphaXiv — refusés avec la justification (pipeline déterministe ≠ recherche exploratoire), pour que la décision survive à la session.

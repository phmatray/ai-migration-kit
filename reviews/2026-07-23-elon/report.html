Majeur Aucune métadonnée de distribution : compatibility, license, metadata.version absents des six skills documentationskills/legacy-upgrade/SKILL.md:2 🔥S

Le kit est public et ses skills ont des exigences d'environnement dures (legacy-upgrade : RoselineMCP + dotnet ≥ 8 + git + python3 ; lifecycle : gh + skills superpowers + code-review). Le guide prévoit le champ compatibility (1-500 caractères) exactement pour cela, plus metadata (author, version) et license. requirements.json couvre la vérification au runtime ; compatibility couvre la déclaration à la distribution (Claude.ai, standard ouvert, autres plateformes) — il manque cette moitié du contrat. ✅ RÉSOLU (2026-07-23, même session) : license: MIT + compatibility (≤ 500 car.) + metadata (author / version 1.6.0 / suite: ai-migration-kit) sur les six skills ; fichier LICENSE (MIT) à la racine ; champ license dans plugin.json ; présence gardée en CI.

→ Ajouter aux six SKILL.md : compatibility (dérivé de requirements.json), metadata.version (aligné sur le CHANGELOG du kit), metadata.author, et license (décision MIT/Apache-2.0 au niveau repo, répercutée dans le frontmatter).

Majeur Cinq skills sur six sans tests de déclenchement testing 🔥M

Le chapitre Testing du guide prescrit des listes should-trigger / should-not-trigger (10-20 requêtes, cible : déclenchement sur 90 % des requêtes pertinentes). La méthode a été appliquée une seule fois — followups, benché skill-creator 16/16 contre 12/16 sans skill — puis plus jamais. Les trois skills importés (create-issue, implement-issue, merge-pr) ont des verbes ultra-génériques (« file an issue », « merge ») : risque réel de sur-déclenchement chez les installeurs, et aucun signal pour le détecter. ✅ RÉSOLU (2026-07-23, même session) : tests/skills/<name>.triggers.md créés pour les six skills (listes should / should-not en anglais, les « should NOT » ciblant les verbes génériques) ; présence et sections non vides gardées en CI par check-frontmatter.py. Reste à exécuter le banc skill-creator sur ces listes à la prochaine modification de description.

→ Créer tests/skills/<name>.triggers.md par skill : liste should-trigger (requêtes évidentes + paraphrases) et should-not-trigger (sujets adjacents). Les passer au banc skill-creator à chaque modification de description — la méthode déjà prouvée sur followups, généralisée.
